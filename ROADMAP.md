#### Где показывать

- **Знаменатель:** активные задачи на день (`getTasksForDay`) + выполненные, засчитанные на этот день (`getDoneTasksForDay` / `getTaskCreditDayKey` из v0.28.1)

- Задачи **без дедлайна** (inbox) — **не** входят в прогресс дня
- Просроченные, закрытые сегодня — в числителе **сегодня** (уже после v0.28.1)
- Пустой день (0 задач) — полоса 0% или нейтральная подпись «Нет задач на день»

### v0.30.1 — черновики форм задач

> Планируемый релиз **v0.30.1** — сразу после [v0.30](#v030--анимации-фона-20), **до** [оптимизации меню](#v031--оптимизация-меню-навигации).

**Проблема:** в **«Новая задача»** (`TaskForm`) и **быстром захвате** (`QuickCaptureModal`) можно заполнить почти всё поля, затем **кликнуть вне окна** (overlay), нажать **Esc** или ✕ — форма закрывается, **весь ввод пропадает**. Для новой задачи дополнительно вызывается `cleanupDraftAttachments` — **черновые фото на диске тоже удаляются**.

Сейчас закрытие идёт через `Modal` → `modal-overlay onClick={onClose}` (`TaskForm.handleCancel` / `QuickCaptureModal.handleClose`).

#### Что сделать

- [ ] **Автосохранение черновика** при изменении полей (debounce ~300 ms):
  - новая задача — `sessionStorage` ключ `planboard-task-draft-new` (или аналог в `%APPDATA%` для Electron)
  - быстрый захват — `planboard-quick-capture-draft`
  - редактирование существующей — опционально отдельный ключ по `taskId` или не трогать (данные уже в `plan.json`)
- [ ] **Восстановление** при повторном открытии формы — подставить title, deadline, проект, notes, вложения *(метаданные; файлы на диске — если `storageTaskId` совпадает)*
- [ ] **Очистка черновика** только после успешного «Сохранить» или явного «Отменить» с подтверждением
- [ ] **Защита от случайного закрытия** — если форма *dirty*:
  - диалог «Есть несохранённые изменения» при клике по overlay / Esc / ✕, **или**
  - отключить `closeOnBackdrop` для форм задач (закрытие только кнопкой «Отмена»)
- [ ] **Быстрый захват** — те же правила; не сбрасывать текст при случайном клику

#### Техника

- `lib/taskDraft.ts` — `saveTaskDraft`, `loadTaskDraft`, `clearTaskDraft`, `isDraftDirty`
- `TaskForm.tsx`, `QuickCaptureModal.tsx` — подписка на поля, restore on mount
- `Modal.tsx` — опция `closeOnBackdrop?: boolean` (default `true`) или `onRequestClose` с возможностью `preventDefault`
- `useHotkeys.ts` — Esc: не закрывать сразу, если dirty → confirm
- `verify-task-draft.mjs` — save/load/clear, не удалять draft при backdrop без подтверждения

**Критерии:**
- [ ] Набрал заголовок + дедлайн + заметку → клик вне модалки → открыл снова — **поля на месте**
- [ ] После «Сохранить» черновик **пустой**
- [ ] Явная «Отмена» с текстом — по желанию сбрасывает черновик (с confirm, если dirty)
- [ ] `pnpm test` — verify-скрипт черновиков

### v0.30.2 — прогресс дня: проценты и подписи

### v0.31.1 — скрываемое меню: режимы навигации

**Связь с minimal-окном:** цепочка фокуса `полный UI` → `скрытый сайдбар` → [minimal Electron](v0291--компактное-минимальное-окно) — три уровня «меньше хрома».

#### Базовый минимум (MVP)

- [ ] Переключатель **«Скрыть меню»** в шапке / `[` (как VS Code) / настройки → Поведение
- [ ] Состояние в `plan.json`: `settings.navigation.sidebarMode`
- [ ] Горячие клавиши `1`–`8` работают и при скрытом меню

#### Варианты — за рамками «стандартного коллапса»

| ID | Режим | Суть | Зачем интересно |
|----|--------|------|-----------------|
| **Peek** ⭐ | **Край-peek** | Сайдбар **overlay** поверх контента; слева тонкая полоска accent (~4px). Наведение / клик по краю — меню выезжает **без сдвига** `main-content` | Максимум места под календарь и повестку; как Edge / Figma |
| **Rail** | **Руническая рейка** | 48px: только иконки вкладок + счётчики-точки; wordmark → эмблема. Разворот по hover или pin | Компактно, но всё на виду; в духе палитр PlanBoard |
| **Palette** | **Круговое меню** | Долгий клик / средняя кнопка на **BrandMark** в шапке — **радиальное** меню вкладок (как HUD в играх) | Фирменный жест; 0px сайдбара постоянно |
| **Command** | **Палитра команд** | `Ctrl+K` — fuzzy-поиск: «повестка», «входящие», действия («новая задача», «экспорт») | Power-user; сайдбар можно не показывать вообще |

| **Float** | **Плавающий док** | Свернутый сайдбар — **плашка** у левого края (blur, тень), перетаскиваемая по Y | Не мешает, но под рукой |

```

├── Встроенные (Классика, Нордскол, … Ведьмак)   ← текущие colorPalette
```

### v0.23 — мобильное приложение

> **Эпик v0.23** → планируемый релиз **[v0.32](#v032--мобильное-приложение)**.

Нужна **мобильная версия** для просмотра и быстрого редактирования плана с телефона. Идеал — **один источник данных** с десктопом; минимум — надёжный **экспорт / импорт** без потери.

#### Варианты интеграции (рекомендация)

| Вариант | Суть | Плюсы | Минусы | Когда выбирать |
|---------|------|-------|--------|----------------|
| **A. PWA** | Адаптивный UI + «Установить на экран» из браузера | Один код с Vite/React, быстрый MVP | Нет доступа к файловой системе ОС; синк только вручную или через облако | Проверить UX на телефоне за 1–2 спринта |
| **B. Capacitor** ⭐ | Обёртка над тем же React-бандлом | ~90% общий код, доступ к файлам / Share API, можно в Store | Нужна отдельная сборка iOS/Android | **Оптимальный баланс** для PlanBoard |
| **C. React Native / Expo** | Отдельное мобильное UI | Нативнее, жесты | Дублирование UI; общее только `types` + селекторы | Если PWA/Capacitor не устроят по UX |
| **D. Только синк-файл** | Десктоп + мобилка читают один `plan.json` в облачной папке | Один файл = один источник правды | Конфликты при одновременном редактировании | Как **фаза 1** параллельно с A или B |

**Рекомендуемый путь:** **B (Capacitor)** + поэтапная синхронизация:

1. **Фаза 1 — файловый обмен (уже почти есть):** импорт/экорт JSON, «Поделиться файлом» на телефоне, опционально путь к `plan.json` в папке OneDrive / Google Drive / iCloud (watch + merge при открытии).
2. **Фаза 2 — WebDAV / облачная папка:** настройка URL + автосинк при старте и по таймеру; стратегия merge: last-write-wins или ручной выбор при конфликте.
3. **Фаза 3 — свой бэкенд (опционально):** лёгкий API + аккаунт; только если нужен real-time между несколькими устройствами.

#### Общий код (monorepo, позже)

```
packages/
  core/          ← types, selectors, exportPlanText, dailyMeetings
apps/
  desktop/       ← Electron (текущий проект)
  mobile/        ← Capacitor shell + общий src или symlink
```

Либо без monorepo на старте: вынести `src/lib/` и `src/types/` в `@planboard/core` позже.

#### Мобильный MVP (v0.23)

- Адаптивная вёрстка: дашборд, повестка, отметка «готово», входящие
- Чтение/запись `plan.json` через Capacitor Filesystem или импорт из «Файлы»
- Те же палитры и темы (упрощённый сайдбар → bottom nav)
- **Не в MVP:** drag на календаре, Jira, Ollama, минималистичное окно Electron

**Критерии:**
- [ ] Приложение открывается на Android *(iOS — по возможности)*
- [ ] План загружается из JSON (файл или импорт)
- [ ] Изменения сохраняются и подхватываются на десктопе через экспорт/импорт или общую папку

### v0.26 — дейлик: группировка похожих задач

> **Эпик v0.26** → планируемый релиз **[v0.33](#v033--дейлик-группировка-похожих-задач)**.

В отчёте «Скопировать для дейлика» и на вкладке **Дейлик** — не выводить подряд десяток почти одинаковых формулировок, а **объединять по смыслу** в одну группу с общим заголовком.

**Пример:** «Поправить баг в логине», «Фикс авторизации», «Login bug» → одна строка: **«Авторизация / логин»** (3 задачи) или список под одним заголовком.

#### Подход (поэтапно)

| Фаза | Как | Зависимости |
|------|-----|-------------|
| **MVP** | Нормализация текста + fuzzy match (Levenshtein, общие токены, тот же проект) | Без ИИ |
| **Улучшение** | Embeddings / запрос к **Ollama** — кластеризация и общий ярлык группы | v0.3 (опционально) |

#### Поведение

- Группировка только в **UI дейлика** и в **тексте для копирования** — исходные задачи в плане не сливаются
- Порог «очень похожие» — настраиваемый (строго / умеренно) или авто с консервативным дефолтом
- Внутри группы — перечисление оригинальных названий или счётчик `(×3)`
- Учитывать **проект**: не смешивать разные проекты без явного совпадения текста
- Ручное «разъединить группу» в отчёте — backlog, не блокер MVP

**Критерии:**
- [ ] В отчёте за период дейлика 5+ похожих задач отображаются компактнее, чем 5 отдельных строк
- [ ] «Скопировать для дейлика» отражает те же группы
- [ ] Логика периода / 13:00 / прошлые дейлики **не ломается** (`verify-daily-meetings.mjs`)
- [ ] Тесты на `groupSimilarDailyTasks()` (или аналог) в `pnpm test`

| **Моя тема** | Наследует палитру-основу (`basedOn`) | Тот же выбор пресетов интенсивности, что у встроенных |

*Не «одна анимация на всех», а **отдельный профиль** в `ambientProfiles.ts` на каждую палитру.*

#### Варианты интенсивности (хочется посмотреть посильнее)

Сейчас в настройках только «По палитре» / «Выключить» — эффект часто **едва заметен**. Нужны явные уровни и возможность **прибавить**:

| Пресет | Суть | Для кого |
|--------|------|----------|
| `off` | Без canvas | Минимализм, слабый ПК |
| `subtle` | Мало частиц, низкая opacity | Фон «намёком» |
| `auto` | Баланс по умолчанию *(текущий дефолт)* | Обычное использование |
| `intense` | **Больше частиц, ярче, крупнее** — «посильнее», но без лагов UI | Хочется «вау»-эффект |
| `max` *(опционально)* | Экспериментальный максимум; предупреждение про CPU | Только для мощных ПК / демо |

**В настройках (таб «Оформление»):**
- Переключатель уровня + **live-превью** на 3–5 с при смене (чтобы сравнить `auto` vs `intense`)
- Подпись: «Интенсивность не влияет на читаемость текста» — glow остаётся на фоне, не на панелях

Поле в `plan.json`: расширить `ambientAnimation` → `'off' | 'subtle' | 'auto' | 'intense' | 'max'` (или отдельный `ambientIntensity`).

#### Техника

- Расширить `AmbientBackground`: новые `kind` частиц (`streak`, `rune`, `fog`, `spark`, `ember`, `nebula`)
- Профили в `ambientProfiles.ts` — **по одному на палитру**, параметры масштабируются множителем интенсивности
- `requestAnimationFrame` + лимит FPS при свёрнутом окне / `document.hidden`
- `prefers-reduced-motion: reduce` → только статичный `--bg-frost`
- Плавный **crossfade** canvas при смене палитры или интенсивности
- Не блокировать UI: один canvas, DPR ≤ 2, пауза вне viewport
- Dev-режим *(опционально)*: `?ambient=intense&palette=starwars` для быстрого просмотра вариантов

**Критерии:**
- [ ] У **каждой** из 6 декоративных палитр (WoW + вселенные) — **свой** эффект, отличимый от остальных на уровне `auto`
- [ ] Режим **`intense`** заметно сильнее `auto` на той же палитре (больше/ярче частиц) — можно сравнить в настройках
- [ ] `plain` без анимации; переключение палитры без фриза >100 ms на типичном ПК
- [ ] `ambientAnimation` / интенсивность сохраняется в `plan.json`, работает с «Моей темой»
- [ ] `verify-ambient.mjs` (или расширение `verify-palettes`) — профиль + множители на каждую палитру
- [ ] Electron + браузер: одинаковое поведение
